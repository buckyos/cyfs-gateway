# 脚本模块化能力 RFC（草案）

- status: Draft
- scope: `src/components/cyfs-process-chain`
- goal: 支持可复用函数/子流程参数化，降低脚本重复，并保持现有执行模型兼容

## 1. 背景

当前脚本复用主要依赖 `exec` 与已有 `invoke` 能力，但仍存在问题：

1. 参数契约不显式，调用方和被调方约定分散在脚本里，易错。
2. 输入输出约定不统一，调用结果与副作用边界不清晰。
3. 跨脚本/跨库复用能力弱，命名冲突与可发现性较差。
4. 控制流边界不够明确，`return/goto` 等动作可能干扰调用方语义。

## 2. 设计目标

1. 在不破坏现有 DSL 的前提下增强复用能力（兼容优先）。
2. 参数化调用具备显式签名、默认值和错误语义。
3. 调用边界清晰：变量作用域、控制流、返回值约定统一。
4. 渐进落地：优先复用 `invoke`，避免引入平行调用机制。

## 3. 非目标

1. 不引入完整编译期类型检查器（后续可结合轻量类型系统）。
2. 不在第一阶段引入复杂依赖管理（版本解算、远程模块拉取等）。
3. 不重写现有 `exec` 语义，仅补足模块化调用能力。

## 4. 总体方案

核心原则：**以 `invoke` 为唯一模块化调用入口**，逐步增强。

1. 增加“可调用单元签名”：
   - 参数名
   - 必填/可选
   - 默认值（可选）
2. 标准化调用帧：
   - 输入：`__args`（只读）
   - 输出：`__ret`（建议使用 Map）
3. 明确控制边界：
   - 被调单元默认不可穿透影响调用方控制流
   - `return/goto/error` 在调用边界内解释
4. 命名空间扩展：
   - 支持 `lib_id::func_id` 形式调用，避免冲突

## 5. 语法草案（MVP）

### 5.1 定义可调用单元（草案）

```xml
<function id="route_by_geo" params="req, fallback=upstreamDefault">
    <![CDATA[
        local ip=$__args.req.clientIp;
        local default_chain=$__args.fallback;

        # ...计算逻辑
        map-create --block __ret;
        map-add __ret target.chain $default_chain;
        return "ok";
    ]]>
</function>
```

说明：

1. `params` 仅为草案格式，最终可改为结构化属性或子节点。
2. `__args` 只读，防止被调流程篡改调用参数。
3. `__ret` 由被调方写入，用于结构化返回。

### 5.2 调用（草案）

```txt
invoke route_by_geo --req $REQ --fallback upstreamA --out geo_result;
```

调用后：

1. 命令返回值仍可参与 `&&` / `||`。
2. `--out geo_result` 将 `__ret` 映射到调用方变量。
3. 调用方可读取 `$geo_result.target.chain`。

## 6. 运行时语义

### 6.1 参数绑定

1. 绑定顺序：显式参数 > 默认值 > 缺失报错。
2. 未声明参数：默认报错（可预留宽松 policy）。
3. 缺失参数错误建议统一错误码：`E_INVOKE_MISSING_ARG`。

### 6.2 调用帧与作用域

1. 每次 `invoke` 建立独立调用帧（至少 block 级隔离）。
2. 被调单元局部变量不泄漏到调用方。
3. `__args`/`__ret` 为调用帧保留变量，避免名称冲突。

### 6.3 控制流边界

1. 被调单元 `return` 只返回给调用方 `invoke`，不直接结束外层链。
2. 被调单元 `error` 只影响 `invoke` 命令结果（除非调用方显式传播）。
3. `goto` 默认禁止跨调用边界跳转；跨边界行为必须显式声明且默认关闭。

## 7. 命名空间与可发现性

建议新增两种调用定位：

1. 同库调用：`invoke func_id ...`
2. 跨库调用：`invoke lib_id::func_id ...`

解析策略建议：

1. 链接阶段构建符号表，优先在本库查找。
2. 冲突时报错并提示候选列表。
3. 保留向后兼容策略：未命中时走旧逻辑。

## 8. 与现有能力关系

1. 与 `exec`：`exec` 保持原语义；模块化场景推荐使用增强后的 `invoke`。
2. 与 `goto`：`goto` 继续服务流程控制，不作为模块化复用主机制。
3. 与 policy：后续可扩展 `ExecutionPolicy`，控制参数缺失/未声明参数等行为。
4. 与类型系统：参数签名可在后续引入轻量类型注解（如 string/map/list/bool）。

## 9. 兼容策略

1. 默认保持旧行为，新增能力需显式使用（`invoke` 新参数或新定义块）。
2. 不改变现有脚本 `invoke` 的最小语义。
3. 新增严格校验可通过 policy 开关启用（先 warn 再 strict）。

## 10. 分阶段落地建议

### P0（最小可用）

1. `invoke` 参数签名（必填/默认值）。
2. `__args` 只读注入，参数缺失统一错误。
3. `--out` 标准化输出映射（Map）。

### P1（边界完善）

1. 调用帧隔离加强（变量泄漏防护）。
2. 控制流边界收敛（return/error/goto 规则固化）。
3. 错误码与定位信息统一（链/块/行/源码片段）。

### P2（跨库模块化）

1. `lib::func` 命名空间调用。
2. 链接期符号校验与冲突检测。
3. 模块索引/帮助文档导出能力。

### P3（类型增强联动）

1. 参数类型声明（可选）。
2. 参数类型转换 policy（strict/warn/legacy）。
3. 结构化返回类型约束（`__ret` schema，可选）。

## 11. 测试矩阵建议

1. 参数绑定：
   - 必填缺失
   - 默认值覆盖
   - 未声明参数
2. 调用边界：
   - 局部变量隔离
   - `__args` 只读
   - `__ret` 输出映射
3. 控制流：
   - 被调 `return/error/goto` 对调用方影响
4. 跨库调用：
   - `lib::func` 正常解析
   - 冲突与未找到错误提示

## 12. 开放问题

1. `function` 定义语法是属性风格还是结构化子节点风格？
2. `--out` 是否允许直接输出到已有非 Map 变量（建议默认禁止）？
3. 未声明参数是否允许透传到 `__args`（建议默认禁止，policy 可放宽）？
4. 是否需要 `invoke --strict` 局部开关覆盖全局 policy？

## 13. 总结

本草案建议以 `invoke` 为核心增强模块化能力，避免引入新调用机制导致语义分裂。优先落地“签名 + 调用帧 + 边界规则”，可在保持兼容的同时显著提升脚本复用和可维护性。
