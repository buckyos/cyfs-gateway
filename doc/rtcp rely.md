
# RTCP Overlay 对等隧道网络设计草案

## 1. 文档目的

本文基于现有 RTCP 设计，整理并扩展出一套面向未来的对等隧道网络模型。目标不是把 RTCP 继续限定为“让 SN 访问 NAT 后 OOD 的工具”，而是把它提升为一个 **TCP First、节点对等、支持透明中转的 Overlay Tunnel Network**。

本文既保留现有设计里最重要的原始动机，也补充了协议层面最好提前明确的边界：哪些能力必须进入协议，哪些策略应留给未来实现自由选择。

---

## 2. 背景与原始问题

RTCP 的原始设计非常纯粹：解决 **NAT 后节点 OOD 无法被主动访问** 的问题。

典型场景如下：

- OOD 位于 NAT 后。
- OOD 可以主动连接 SN。
- SN 无法主动连接 OOD。
- 业务上却希望 SN 在需要时可以访问 OOD 上的服务，例如 HTTP 服务。

现有思路是：

1. OOD 主动与 SN 建立一条 **Keep Tunnel**。
2. Tunnel 维持长连接。
3. SN 收到外部 HTTP 请求后，在这条 Tunnel 上执行 `OpenStream`。
4. OOD 侧把该 Stream 映射到本地 HTTP 端口。
5. 从业务效果上看，SN 就像“主动连接”到了 OOD。

其本质并不是让 SN 真正穿透 NAT 去主动拨号，而是：

> 复用由 OOD 主动建立的长连接，在其上打开新的逻辑流，从而获得“可被访问”的效果。

这就是 RTCP 的原始价值：

> **用一条保持在线的 Tunnel，把原本单向可达变成逻辑上的双向可达。**

---

## 3. 架构演进的动机

新的需求来自用户在外出差的场景：

- 用户在另一个 NAT 后使用一台 Laptop。
- 该 Laptop 希望访问家中的 OOD。
- 更重要的是，系统并不希望 Laptop 只是一个“普通客户端”。

如果 RTCP 的核心理念是 **节点对等**，那么当 Laptop 加入网络并建立了某种 Tunnel 之后，它在逻辑上也应该和 OOD 一样，成为一个完整节点：

- Laptop 可以主动访问 OOD。
- OOD 在需要时也可以主动访问 Laptop。

也就是说，RTCP 不应只停留在：

`Client -> SN -> OOD`

而应该提升为：

`Laptop <-> OOD`

中间是否经过中转，不应改变这条关系的对等性。

因此，RTCP 的设计目标开始从：

> NAT 穿透 / 可达性修复

演进为：

> **对等节点之间的逻辑隧道网络**

---

## 4. 设计目标

本文整理出的目标如下。

### 4.1 节点对等

RTCP 协议层不应固化“客户端/服务端”身份。建立 Tunnel 时可能存在发起方和接收方，但 Tunnel 一旦建立成功，双方都应视为对等节点，均可：

- OpenStream
- 发起请求
- 承载服务
- 在授权前提下被对方访问

### 4.2 TCP First

考虑到当前互联网环境中 UDP P2P 的稳定性显著下降，协议应优先依赖 **TCP 长连接** 建立稳定网络。UDP 可保留为未来扩展方向，但不应成为当前架构的前提条件。

### 4.3 透明中转

协议需要支持一种中转方式：

- 中转节点不理解上层业务协议。
- 中转节点不负责 HTTP/自定义业务协议转换。
- 中转节点只负责帮助构造或转发 Tunnel / Stream。

这样，端点看到的仍是一条标准 Tunnel，中途是否经过 Relay 不改变其点对点语义。

### 4.4 协议只定义能力，不定义唯一策略

P2P/Overlay 网络一旦部署，协议后续很难大幅修改。因此协议设计应尽量遵循：

> **协议提供机制（mechanism），实现决定策略（policy）。**

具体到本文语境中，这意味着协议不能过早把“路由一定由谁选择”写死。协议应该允许：

- 中转节点自己选路径。
- 客户端指定路径。
- 未来演进为更复杂的混合方式。

### 4.5 支持从单跳扩展到多跳

从实现复杂度出发，初期实现可以只做一跳透明中转；但从协议演进角度，最好不要把自己锁死在“一跳”这个限制上。协议应对 **N 跳透明中转** 保持开放。

---

## 5. 非目标与边界

为了防止设计范围失控，本文明确以下内容暂不强行纳入本轮协议核心：

1. **不要求定义完整的全网路由算法。**
   协议只需要允许路径被表达、被提示、被选择，并不要求本轮就定义全网最优路由。

2. **不要求强制所有节点都开放中转能力。**
   所有 RTCP Stack 在理论上都具备成为 Relay 的能力，但是否对外开放为 Relay，属于节点本地策略。

3. **不要求与显式业务网关合并。**
   显式中转和透明中转可以共存，它们解决的是不同层次的问题。

## 6. 核心抽象与术语

为了减少后续讨论歧义，建议在文档层面对几个概念做严格区分。

### 6.1 Node

任何运行 RTCP Stack 的实体都称为 Node，例如：

- OOD
- SN
- Laptop
- 未来的其他在线节点

### 6.2 Underlay Tunnel（基础 Tunnel）

相邻两个节点之间直接维持的 RTCP Tunnel。它通常对应一条真实的 TCP 长连接或由其承载的稳定会话。

建议保留原有原则：

> **同一对相邻节点在同一时刻只保持一条活跃基础 Tunnel。**

这样可简化：

- 会话管理
- 重连策略
- Stream 复用
- 路由状态维护

### 6.3 Overlay Tunnel（逻辑 Tunnel）

端到端节点之间的逻辑 Tunnel。它看起来是 Node A 与 Node B 之间的一条标准 Tunnel，但底层可能实际经过多个 Relay 节点。

例如：

`Laptop -> SN -> OOD`

在协议语义上表现为：

`Laptop <-> OOD`

### 6.4 Stream

Tunnel 上承载的逻辑流。上层业务数据（HTTP、自定义 RPC、文件传输等）应尽可能运行在 Stream 上，而不是要求为每个业务重新建连接。

### 6.5 Explicit Relay（显式中转）

中转节点理解上层业务协议，并对其进行转换或代理。

例如：

- Browser 把 HTTP 请求发给 SN。
- SN 理解 HTTP 业务。
- SN 把请求转成 RTCP / Stream 送给 OOD。

这是业务层网关行为。

### 6.6 Transparent Relay（透明中转）

中转节点不理解上层业务语义，只帮助构造或转发 Tunnel/Stream。端点感知到的仍是一条点对点 Tunnel。

### 6.7 Relay Node

参与透明中转路径构造或数据转发的节点。SN 是天然的候选 Relay，但不应是协议上唯一的 Relay 角色。

## 7. 对等模型

RTCP 的一个关键设计思想应被明确写进文档：

> **协议栈视角下没有永久性的“服务器”概念，只有当前谁在发起、谁在接收、谁在帮忙中转。**

换句话说：

- OOD 可以是服务提供方。
- Laptop 也可以是服务提供方。
- SN 可以是中转方。
- 同一个节点在不同会话中可以扮演不同角色。

一旦某条 Overlay Tunnel 建立成功，两端应当都获得“把对方视为 Server”的能力。是否真正允许发起访问，取决于授权与策略，而不是 Tunnel 模型本身。

---

## 8. 显式中转与透明中转的区分

这部分是整个设计里非常重要的一条边界。

### 8.1 显式中转

显式中转的本质是：

> 中转节点理解业务，并以业务逻辑决定如何转发。

例如：

`Laptop/Browser -> HTTP -> SN -> RTCP -> OOD`

SN 在这里不仅是网络中转点，也是业务协议转换点。

### 8.2 透明中转

透明中转的本质是：

> 中转节点不理解业务，只负责让两端像直接连通一样工作。

例如：

`Laptop -> RTCP -> SN -> RTCP -> OOD`

但对两端来说，看到的仍是：

`Laptop <-> OOD`

### 8.3 两者的关系

两者不是互斥关系，而是不同层次能力：

- 显式中转属于 **接入/业务网关能力**。
- 透明中转属于 **传输/Overlay 网络能力**。

现有 HTTP Upstream 方案可以继续保留；而透明中转则为 CLI、自定义客户端、对等服务调用提供统一底座。

---

## 9. 一跳与多跳

### 9.1 一跳透明中转

最简单的形式是：

`Laptop -> SN -> OOD`

其优点是：

- 实现简单
- 状态少
- 容易调试

不足是：

- 只要唯一 Relay 不可达，连接就失败
- 灵活性较差

### 9.2 多跳透明中转

更一般的形式是：

`Laptop -> SN1 -> SN2 -> ... -> OOD`

这使 RTCP 逐步变成一个真正的 Overlay Network。多跳能力有几个明显价值：

- 提高复杂网络环境下的可达性
- 允许绕过某些不可直达节点
- 为后续自治路由/分层中继/区域边界节点提供空间

### 9.3 设计建议

比较稳妥的路径是：

- **协议层面支持多跳表达。**
- **实现层面允许先只落地一跳。**

这样既不牺牲长期演进空间，也不会把第一版实现做得过重。

---

## 10. 路径构造的两种模式

围绕“谁负责选路”，至少存在两种合法模式。

### 10.1 Relay 自主选路

源节点向某个 Relay 发起请求：

> 请帮我建立一条到目标节点的 Tunnel。

Relay 的职责是：

- 决定是否愿意承担中转
- 决定下一跳是谁
- 决定是否继续递归寻找后继 Relay

源节点不关心中间细节，只关心最终是否建立成功。

这种模式的好处是客户端简单，网络自治能力强。

### 10.2 客户端指定路径

另一种方式是，Relay 先把自己所知的候选路径或候选下一跳返回给源节点，源节点再明确指定：

- 选哪条路径
- 允许经过哪些节点
- 最大跳数是多少

Relay 只负责 **精确执行**。

这种模式的好处是：

- 路径选择权在发起方
- 便于后续做策略优化
- 便于上层系统加入拓扑认知

### 10.3 推荐结论

从协议设计角度看，最优解不是二选一，而是：

> **两种模式同时支持。**

因为协议一旦上线，再想补回被删掉的选择权会非常困难。

## 11. 协议扩展建议

这一部分是本文最核心的补充建议。目标是用尽量少的命令，把未来的选择权保留住。

### 11.1 能力声明（推荐）

节点建立基础 Tunnel 后，建议交换能力信息，例如：

- `relay_capable`: 是否具备对外中转能力
- `supports_route_hint`: 是否支持返回候选路径/候选下一跳
- `supports_strict_path`: 是否支持严格按指定路径执行
- `max_relay_hops`: 本节点愿意承担的最大中转深度
- `supports_overlay_tunnel`: 是否支持透明 Overlay Tunnel

这样可以把“协议能力”和“本地策略”分离开。

### 11.2 RelayTunnelRequest

建议增加一个面向透明中转的核心命令，例如可命名为：

`RelayTunnelRequest`

其最小语义是：

> 请求接收方帮助建立一条通往目标节点的 Overlay Tunnel。

建议字段：

- `target_node_id`
- `path_mode`：`auto` / `strict`
- `path`：可选，节点列表或路径描述
- `max_hops`：可选
- `purpose`：可选，表明这条建立请求是业务访问还是中转基础设施用途
- `metadata`：预留扩展字段

其中：

- `path_mode=auto` 表示由中转节点自行选路。
- `path_mode=strict` 且 `path` 已填写时，表示请求方显式指定路径。

### 11.3 RelayTunnelResponse

接收方收到请求后，至少应支持三类响应。

#### A. ACCEPTED

语义：

> 我接受这个请求，并负责为你建立这条 Overlay Tunnel。

这对应你原始想法中的“OK，我可以帮你打开”。

一旦返回 ACCEPTED，请求方即可把后续流量发给该入口，由对方负责完成后续建立和转发。

#### B. ROUTE_HINT

语义：

> 我当前不能直接按默认方式替你建立，但我知道若干可行候选路径或候选下一跳。

返回内容可以是：

- 候选路径列表
- 候选下一跳列表
- 跳数/代价等简单辅助信息

注意，这里不一定要暴露全网完整拓扑。为保护网络隐私和控制信息规模，候选路径既可以是明确的节点序列，也可以是较为抽象的“路径句柄”。

#### C. REJECTED / UNREACHABLE

语义：

> 我不能为该目标建立 Tunnel，也没有合适候选可返回。

建议附带原因码，例如：

- 不支持该能力
- 本地策略拒绝
- 超出跳数限制
- 目标不可达
- 指定路径非法或不可执行

### 11.4 严格路径语义

这一点建议明确写成协议级强约束：

> **当 `path_mode=strict` 且 `path` 被明确填写后，接收请求的 Relay 节点必须严格按照该路径执行；若无法执行，应直接失败，而不是私自改路。**

这是因为一旦允许 Relay 偷偷改路，就等于把“客户端选路”这件事在协议层做空了。

### 11.5 递归执行

当 Relay 自主选路时，允许其对下一跳递归发起同类请求。也就是说：

- 上层请求语义保持不变
- 每一层都在回答同一个问题：
  - 我能否直接建立？
  - 不能的话，我能否推荐或递归寻找下一跳？

这使协议天然支持从一跳扩展到 N 跳，而不需要重新发明一套新命令。

## 12. 建议的状态机

为了方便实现，建议把透明中转相关流程抽象为以下状态：

1. `INIT`
2. `RESOLVING`
3. `ACCEPTED`
4. `ESTABLISHING`
5. `ESTABLISHED`
6. `FAILED`
7. `CLOSING`

其中：

- `INIT -> RESOLVING`：收到 RelayTunnelRequest
- `RESOLVING -> ACCEPTED`：节点决定接管建立
- `RESOLVING -> FAILED`：无法建立且无候选
- `RESOLVING -> INIT/WAIT_CLIENT`：返回 ROUTE_HINT，等待上层再次发起指定路径请求
- `ACCEPTED -> ESTABLISHING`：开始逐跳建立
- `ESTABLISHING -> ESTABLISHED`：端到端 Overlay Tunnel 建成
- 任意状态在遇到错误时进入 `FAILED` 或 `CLOSING`

这部分不一定要写成特别重的 RFC 状态机，但尽早统一术语会明显降低实现歧义。

## 13. 中转能力的开放策略

虽然“所有 RTCP Stack 理论上都拥有成为 Relay 的能力”是很自然的结论，但这件事最好拆成两层：

1. **协议能力层面**：任何节点都可以实现 Relay 相关命令。
2. **策略开放层面**：节点是否愿意对外承担 Relay，要由本地策略决定。

这样做的好处是：

- 不会把边缘设备强制变成公共中继
- 不会把协议能力和部署策略混为一谈
- 未来可以自然引入专用 Relay、家庭设备 Relay、移动节点 Relay 等角色

## 14. 安全与权限建议

这部分虽然不是你口述的重点，但从设计文档角度最好明确写出来，否则后面一定会回头补。

### 14.1 认证

无论是基础 Tunnel 还是 Overlay Tunnel，节点身份都必须是可认证的。否则无法定义“允许谁向谁发起请求”。

### 14.2 授权

对等性并不意味着默认无条件互访。Tunnel 建立成功后，是否允许一方主动访问另一方服务，仍应受授权策略控制。

也就是说：

- “可达”不等于“可访问全部能力”。
- “节点平等”不等于“权限平等”。

### 14.3 中继透明性与加密

既然透明中转强调中继节点不理解业务，那么从长期看，最好支持 **端到端业务流加密**。这样 Relay 只看到必要的 RTCP 控制信息，不需要感知上层载荷内容。

### 14.4 拓扑暴露控制

如果 Relay 能返回候选路径，建议控制拓扑暴露粒度，避免把整个网络结构无条件泄露给任意节点。必要时可以考虑：

- 只返回候选下一跳
- 返回路径句柄而非全路径
- 对不同信任级别返回不同精度的路由提示

## 15. 与现有系统的关系

本文方案不是推翻现有系统，而是将其分层化。

### 15.1 原有模型仍然成立

现有场景：

- OOD 主动连 SN
- SN 收到 HTTP 请求
- SN 通过现有 Tunnel `OpenStream` 到 OOD

这一模型仍然是合理的，而且在浏览器入口场景下非常有效。

### 15.2 新方案补足的是“对等节点互联”

透明中转新增的是：

- Laptop 这种外部设备不再只是浏览器式客户端
- CLI / 自定义客户端获得一等公民地位
- OOD 与 Laptop 可以形成标准端到端 Tunnel

### 15.3 SN 的角色变化

SN 不再只是一台“业务入口服务器”，而更像一个：

- 稳定在线节点
- Rendezvous / Relay 候选节点
- Overlay 网络中的高可用骨干点

但协议层不应把这种角色专属化。SN 很重要，但不应成为唯一合法中继。

---

## 16. 一个比较稳妥的落地路线

从工程上讲，建议分阶段推进。

### Phase 1：统一抽象

先把以下概念在代码和文档里统一：

- Node
- Underlay Tunnel
- Overlay Tunnel
- Transparent Relay
- Explicit Relay

这是后续一切讨论的前提。

### Phase 2：做一跳透明中转

先支持：

- Laptop <-> SN 基础 Tunnel
- SN <-> OOD 基础 Tunnel
- 由 SN 帮助建立 Laptop <-> OOD 的 Overlay Tunnel

同时把协议字段中的 `path`、`path_mode`、`max_hops` 先保留下来，即使第一版只支持最简单的情况。

### Phase 3：引入 ROUTE_HINT

让 Relay 在不能直接完成时，可返回候选路径/下一跳信息，为客户端选路留接口。

### Phase 4：引入递归多跳

在不改变核心命令的前提下，允许 Relay 对下一跳递归发起同类请求，把网络自然扩展到多跳。

这条路线的好处是：

- 第一版实现压力可控
- 协议不会被一跳限制锁死
- 演进路径清晰

## 17. 仍待进一步收敛的问题

以下问题已经浮现，但本轮可以先作为开放问题保留：

1. **Overlay Tunnel ID 如何分配与追踪**
2. **端到端 Stream ID 与逐跳 Stream ID 的映射方式**
3. **中继失败后的重试和路径切换策略**
4. **路径代价的度量方式（跳数、延迟、稳定性、信任等级）**
5. **Topology Hint 的最小可行格式**
6. **节点发现机制是否单独定义，还是依赖外部控制面**
7. **是否支持把“为中转目的建立的 Tunnel”显式区分为基础设施用途**

这些问题值得继续讨论，但不影响当前主干抽象成立。

---

## 18. 总结

把前面的讨论收束成一句话，RTCP 的演进方向可以定义为：

> **RTCP 不再只是一个帮助 SN 访问 NAT 后 OOD 的协议，而是一个基于 TCP 长连接的、支持透明中转的、节点对等的 Overlay Tunnel Network。**

它的关键设计原则有四条：

1. **节点对等**：任何建成 Tunnel 的节点都可以在逻辑上视对方为可服务节点。
2. **TCP First**：优先保证连接可达与稳定。
3. **透明中转**：中继不改变点对点 Tunnel 语义。
4. **机制与策略分离**：协议保留未来选路自由，而不是过早固化唯一方案。

在这个框架下：

- 原有 OOD 场景仍是合法特例。
- Laptop/CLI 场景成为自然扩展。
- SN 从单纯入口节点升级为 Overlay 网络中的稳定 Relay 候选。
- 多跳网络、客户端选路、递归中转都可以在同一套协议框架里逐步演进。

这也是本文最想保住的一点：

> **让协议尽量少做未来的决定，但把未来需要的选择权都留在协议里。**
