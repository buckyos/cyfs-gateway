# Process Chain 语言核心模型概述

本文尝试从 `cyfs_gateway` 的真实使用场景出发，给 `process_chain` 抽象出一个更稳定的语言核心模型。

目标不是重复命令参考，也不是逐条描述当前实现细节，而是回答下面几个更基础的问题：

- 这门语言要解决什么问题。
- 它在网关体系里应该扮演什么角色。
- 哪些概念是一等核心概念，哪些只是实现或运行时细节。
- 后续设计和扩展应该尽量围绕什么边界展开。

本文会参考以下材料，但不完全等同于其中任何一篇旧文档：

- `doc/编写复杂的process_chain.md`
- `doc/process chain env.md`
- `src/components/cyfs-process-chain/doc/ARCHITECTURE.md`
- `src/components/cyfs-process-chain/doc/COLLECTION_BACKEND_ASSESSMENT.md`

## 1. 设计目标

`process_chain` 的设计目的，不是做一门通用脚本语言，也不是做一个数据库查询语言，而是：

- 作为嵌入在 gateway 运行时中的规则与编排语言。
- 用统一方式描述各种网络协议入口上的判断、变换、路由、准入与后处理逻辑。
- 允许系统生成规则、模板化规则和人工运维规则共存。
- 在不重编译宿主服务的前提下，表达可组合、可观测、可替换的业务逻辑。

因此它最自然的定位是：

> 一门面向网关运行时的、可嵌入的、结构化的控制与变换脚本语言。

这里有三个关键词：

- 可嵌入：语言本身不拥有完整世界，它运行在宿主提供的变量、命令和 hook point 中。
- 结构化：它不是任意跳转脚本，而是围绕 block、chain、hook point 组织控制流。
- 控制与变换：它的核心工作不是数值计算，而是读上下文、改上下文、做判断、给出动作。

## 2. 宿主视角下的语言定位

从 gateway 宿主来看，`process_chain` 更像一个“规则执行容器”，而不是一段独立程序。

宿主负责：

- 决定有哪些 hook point。
- 决定每个 hook point 注入哪些环境变量。
- 决定可以调用哪些 external command。
- 决定如何解释脚本最终产出的控制结果。

脚本负责：

- 读取宿主注入的上下文。
- 对请求、响应或连接元数据做判断和加工。
- 调用内置或外部能力。
- 产出结构化控制结果，让宿主继续后续处理。

因此语言和宿主之间的关系应该是：

- 宿主提供能力边界。
- 语言提供规则表达能力。
- 双方通过环境变量、集合、外部命令和控制结果交互。

## 3. 一等核心概念

如果要把 `process_chain` 收束成稳定的语言模型，我认为以下概念应该被视为一等公民。

### 3.1 Hook Point

`hook point` 是语言与宿主交互的第一边界。

它代表：

- 一个明确的执行时机。
- 一组由宿主定义的输入变量。
- 一套约定好的结果解释方式。

例如：

- HTTP pre-request
- HTTP post-response
- DNS request
- SOCKS request
- TUN/RTCP/QUIC 连接建立阶段

因此，`hook point` 不是单纯的“脚本挂载点”，而是一个语义完整的执行上下文。

### 3.2 Chain

`chain` 是一个完整的规则执行体。

它更接近：

- 一次独立执行流程
- 一个可被调用的规则单元
- 一个有局部状态边界的处理过程

在语言模型里，`chain` 最适合被理解为：

> 面向某个任务目标的完整处理过程。

比如：

- 解析 appid
- 执行准入检查
- 选择上游目标
- 对响应做统一后处理

### 3.3 Block

`block` 是 `chain` 内部的结构化步骤单元。

它最合适的语义不是“裸标签”，而是：

- 一个共享同一 chain 局部环境的步骤块
- 一个适合模块化复用的局部规则片段
- 一个结构化控制转移的目标单元

在旧文档里它被类比为 function/sub，这个类比有一定帮助，但不够准确。更准确的说法应该是：

> block 是 chain 内部可命名、可组织、可复用的执行步骤，而不是完全独立的函数。

因为它与普通函数的一个关键区别是：

- 它默认共享 chain 的状态上下文，而不是强隔离调用栈。

### 3.4 Env

环境变量系统是 `process_chain` 的核心数据平面。

语言里的绝大多数逻辑，本质上都在围绕 env 做：

- 读取输入上下文
- 推导中间状态
- 写出处理结果
- 与宿主或后续 block 共享状态

因此 env 不是附属机制，而是语言的主存储模型。

### 3.5 Result / Control

脚本的输出不应该只被理解为“命令成功或失败”，而应该理解为结构化控制结果。

当前语言里已经逐步形成这类模型：

- success value
- error value
- return / error / exit / break
- `match-result` 对结果类型进行分支

这说明 `process_chain` 的核心不是 shell 风格 exit code，而是：

> 用结构化结果驱动宿主和脚本的后续控制流。

## 4. 语言的主执行模型

站在语言设计角度，我建议把 `process_chain` 的主执行模型描述为：

> 在某个 hook point 中，宿主注入初始上下文；脚本在一个结构化执行框架里，对上下文做读写和控制转移，最终产生一个可被宿主解释的结构化结果。

这个模型可以拆成四层。

### 4.1 输入层

输入层来自宿主：

- `REQ`
- `RESP`
- 连接元数据
- 设备信息
- 系统配置映射
- 外部命令能力

这一层的核心特征是：

- 脚本不拥有输入定义权。
- 宿主决定哪些数据可读、哪些可写、哪些只读 visitor 暴露。

### 4.2 执行层

执行层由 `hook point -> chain -> block -> statement` 构成。

它体现的是结构化执行顺序：

- hook point 选择要运行的一组 chain
- chain 负责一个完整处理目标
- block 负责步骤化组织
- statement 负责原子判断与操作

### 4.3 状态层

状态层由 env 和 collection 构成。

它的核心作用是：

- 保存从输入中抽取出的中间语义
- 跨 block 传递已计算结果
- 支撑集合型规则匹配和路由决策
- 为宿主返回最终动作和附加信息

### 4.4 输出层

输出层不是单一字符串，而是控制意图。

它可能表现为：

- 接受请求
- 拒绝请求
- 转发到目标
- 直接返回响应
- 结束当前 block/chain/lib
- 把结果交给宿主继续解释

因此语言的输出层本质上是动作与控制，而不是“打印结果”。

## 5. 推荐的核心语义理解

如果要让用户形成稳定心智模型，我认为应该把这门语言概括成下面这句话：

> `process_chain` 是一门围绕“上下文 -> 判断/变换 -> 控制结果”设计的网关规则语言。

这句话里的三个阶段对应三种一等语义。

### 5.1 上下文

脚本首先面对的是上下文，而不是抽象变量。

典型上下文包括：

- 请求头
- URL
- 来源地址
- 目标服务信息
- 认证状态
- 系统策略数据

所以变量系统的核心职责不是替代编程语言里的局部变量，而是：

- 将宿主输入建模成统一可访问上下文。

### 5.2 判断与变换

脚本的主要工作是：

- 判断当前上下文满足什么条件。
- 把上下文转换成更适合后续使用的状态。
- 组织多个处理步骤之间的共享状态。

也就是说，它不是以“算法”为中心，而是以“规则判断与上下文重写”为中心。

### 5.3 控制结果

脚本最终一定要给出控制意义。

这个控制意义可能是：

- 放行
- 拒绝
- 转发
- 改写后继续
- 调用另一规则再决策

因此控制流语义应该始终服务于“产生动作”，而不是为了语言炫技而增加复杂控制结构。

## 6. 集合在语言中的定位

从语言设计上，我建议把 collection 定位为：

> 规则执行期的结构化状态容器，而不是数据库抽象。

这样有几个好处：

- 能解释为什么当前主模型是 memory-first。
- 能解释为什么 JSON/SQLite 不应该直接定义语言主语义。
- 能解释为什么遍历顺序、引用语义、写保护都比“后端是什么”更重要。

换句话说：

- collection 首先服务于规则表达。
- 持久化只是 runtime backend 的附加能力。
- 什么时候 flush、是否 live update，应该首先看宿主需求，而不是语言本体。

## 7. external command 的正确定位

`external command` 不应该被理解为“语言能力不够，所以开了个逃生窗”，而应该被视为语言的能力扩展机制。

它的合理定位是：

- 语言负责结构化编排。
- 宿主或扩展负责提供协议、系统、加密、查询等专用能力。

例如：

- JWT 校验
- Cookie 解析
- 服务发现
- 内核配置读取
- 协议探测

这些能力本来就不适合全部内建进 DSL 核心。

因此 `process_chain` 更像：

- 一个可组合的规则壳层
- 加上一组可插拔的能力接口

## 8. 模块化的推荐理解

旧文档里把 block、chain 类比成 sub/process，这个方向是对的，但还可以更精炼一些。

推荐这样理解：

- hook point：宿主定义的规则入口
- chain：一个完整职责单元
- block：chain 内的步骤模块
- command：可复用的原子能力
- env/collection：共享状态平面

按这个模型，复杂 process chain 的模块化应优先追求：

- 职责切分清晰
- 状态流动显式
- 控制出口统一

而不是简单把更多逻辑拆到更多 block 里。

## 9. 核心设计原则

如果以后要继续扩展语言，我建议围绕以下原则。

### 9.1 先结构化，再语法糖

先确定核心执行语义，再决定是否提供语法糖。

例如：

- `for`
- `match-result`
- `invoke`
- `goto`

都应该先有明确的控制模型，再讨论写法是否更简洁。

### 9.2 先控制语义一致，再扩更多命令

语言最容易失控的地方不是命令多，而是：

- 同一个问题能用三套控制方式表达。

所以未来更重要的是收敛控制语义，而不是持续增加特殊命令。

### 9.3 先区分语言核心与 runtime 特性

以下内容应尽量视为 runtime 能力，而不是语言核心：

- 持久化 backend
- 宿主注入变量集合
- external command 具体列表
- 某些协议栈的专用 visitor

语言核心只应该关心：

- 如何访问上下文
- 如何组织步骤
- 如何表达判断与控制

### 9.4 先保证可预测，再追求灵活

网关规则语言最重要的品质不是自由，而是：

- 可预测
- 易审计
- 易排障
- 易热更新

因此任何新功能都应该优先评估：

- 会不会让脚本行为更难推理。
- 会不会让规则顺序更难解释。
- 会不会让宿主和脚本的边界更模糊。

## 10. 非目标

为了避免语言越长越像“万能脚本”，也建议明确一些非目标。

`process_chain` 不应以这些目标为优先：

- 通用计算语言
- 完整数据库编程接口
- 大规模数据处理 DSL
- 任意控制跳转语言
- 宿主无关的可移植脚本平台

它首先还是网关场景下的规则编排语言。

## 11. 一句话定义

如果要用一句话概括这门语言，我建议定义为：

> `process_chain` 是一门嵌入在 gateway 运行时中的结构化规则语言，用于在明确的 hook point 上读取上下文、组织步骤、变换状态并产出控制结果。

这个定义比“脚本语言”“流程语言”“配置语言”都更准确，因为它同时涵盖了：

- 宿主嵌入性
- 结构化执行
- 状态变换
- 控制输出

## 12. 对后续设计的直接启发

如果接受上述核心模型，那么后续所有语言设计问题都可以回到几个判断标准：

1. 这个功能是在增强“上下文 -> 判断/变换 -> 控制结果”主链路，还是在引入旁路复杂度。
2. 这个功能属于语言核心，还是属于 runtime/backend/宿主注入能力。
3. 这个功能是在增强结构化表达，还是在放大历史兼容语义。
4. 这个功能会不会削弱可预测性、可审计性和可排障性。

这几个判断标准如果能保持稳定，`process_chain` 后续即使继续演进，也不容易失去语言中心模型。
