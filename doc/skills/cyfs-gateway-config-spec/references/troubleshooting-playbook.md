# 排障手册

本文件定义组网类问题的推荐排障顺序。

## 1. 总原则

不要一上来改 YAML。
先把问题分层：

1. underlay 是否建立
2. `tun` 地址是否可达
3. 路由是否正确
4. DNS / 代理 / 服务发现是否正确
5. 应用层协议是否正确

## 2. 适用于“虚拟局域网 / 显式地址互通”的五层排查

### 第一层：underlay

检查：

- 监听端口是否成功打开
- 两端或多端是否能建立底层连接
- TLS / QUIC / RTCP 所需证书、密钥、名称是否完整

如果 underlay 都没建立，不要先讨论 `tun` 地址或代理规则。

### 第二层：`tun` 地址

检查：

- `tun` 地址是否配置到预期子网
- 子网掩码是否一致
- MTU 是否合理

如果 `tun` 地址冲突或掩码错误，应用通常表现为“偶发可达”或“只能单向通信”。

### 第三层：路由

检查：

- 访问目标地址时是否真的走了 `tun` 路径
- 是否缺静态路由
- 是否把物理网卡地址误当成 `tun` 地址使用
- 若是代理方案，app 是否真的把流量送进 socks 服务

这是“ping 一边通、一边不通”与“代理配置了但没走代理”的高频原因。

### 第四层：DNS / 代理 / 服务发现

检查：

- 应用使用的是固定地址、域名还是代理
- 名称是否解析到 `tun` 地址
- 代理目标 `target` 是否正确
- `rule_config` 是否真的命中了预期流量
- 应用是否依赖广播 / mDNS / NetBIOS

如果依赖广播发现，即使三层可达，也可能表现为“看不见设备”。

### 第五层：应用

检查：

- 目标端口是否监听
- 协议是否允许跨网段 / 跨 NAT / 跨接口
- 应用是否把来源地址校验死在物理网段
- 应用是否根本不支持代理

## 3. 典型故障模式

### 隧道建立了，但 ping 不通

优先检查：

- `tun` 地址是否在同一规划中
- 子网掩码是否一致
- 路由是否进入 `tun`
- 本地防火墙是否丢弃 ICMP

### ping 通了，但 app 不通

优先检查：

- 业务端口是否监听
- 业务端口是否放行
- 应用是否访问了错误地址
- 应用是否依赖广播发现

### 代理配置了，但 app 仍然直连失败

优先检查：

- app 是否真的支持 SOCKS5 或系统代理
- 系统代理是否生效到了该 app
- `target` 是否可达
- `rule_config` 是否命中了预期流量

### DNS 正确，但连接仍失败

优先检查：

- 解析出的地址是否真的可达
- 目标服务是否绑定在 `tun` 地址对应接口
- 应用是否把同名主机缓存到了旧地址

### 小包通，大包不通

优先检查：

- MTU
- 分片
- 封装层额外开销

## 4. 推荐输出模板

排障答案优先按以下格式组织：

1. 先判断故障所在层
2. 再给最小验证动作
3. 再决定要不要改配置

示例结构：

- 当前最可能卡在第 N 层
- 先验证 A、B、C
- 若 A 失败，回到 underlay
- 若 A 成功但 B 失败，排查 `tun` 地址 / 路由 / 代理
- 若三层都正常，再看 DNS / 应用发现机制
