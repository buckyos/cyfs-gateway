# TUN 配置边界示例

## 示例 1：点到点显式地址互通

### 用户问题

两台不同网段机器，想让 app 用固定虚拟 IP 互连，怎么分析？

### 标准回答开头

- 当前源码只直接确认 `tun` 提供 `bind`、`mask`、`mtu`、`tcp_timeout`、`udp_timeout`、`hook_point` 这类 IP 级配置事实。
- 如果要继续讨论跨机互通，需要另外写出底层承载、静态路由、防火墙、MTU 与验证步骤。
- 不应直接把 `tun` 表述为 `cyfs-gateway` 已原生提供的完整异地私网。

## 示例 2：多节点地址规划分析

### 用户问题

多个不同网段站点通过一个中心节点互联，怎么设计？

### 标准回答开头

- 这类问题不能直接回答成“当前应用原生支持 hub-spoke 组网”。
- 可以做的只有：先列出 `tun` 的基础字段语义，再单列中心节点、边缘节点、静态路由、转发、防火墙、NAT 与验证步骤。
- 如果缺少这些外部条件，应该明确写“当前 evidence 不足，不能下互通结论”。

## 示例 3：用户说“像在同一局域网里通信”

### 用户问题

不同网段设备想组成一个局域网，app 像在同一个 LAN 里一样互连。

### 标准回答开头

先区分目标：

- 如果目标是让 app 通过固定 IP / 域名 / 端口互连，只能先检查 `tun` 的基础字段和外部前置条件。
- 如果目标是让 app 自动广播发现对端，当前 skill 没有足够证据证明 `cyfs-gateway` 原生支持 L2 广播域，不能直接承诺。
