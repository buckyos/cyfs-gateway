# 宿主机网络前置条件

本文件用于提醒回答者：很多“cyfs-gateway 方案是否可用”的问题，不只取决于 YAML，还取决于宿主机网络。

## 1. 必须单独列出的宿主机动作

回答跨网段互联、显式地址互通、`tun` 场景时，至少检查并写出：

- 是否需要开启 IP 转发
- 是否需要增加静态路由
- 是否需要放行防火墙规则
- 是否涉及 NAT 或禁止 NAT
- MTU 是否需要调小
- 是否存在与现有网段冲突的 `tun` 子网

不要把这些动作混进“cyfs-gateway 自身配置”里。

## 2. 地址规划检查

先检查：

- `tun` 子网是否与本地物理网段冲突
- 不同节点是否拿到了唯一地址
- 应用是否应访问 `tun` 地址，而不是物理网卡地址

如果这些没讲清楚，就不能说方案“可用”。

## 3. 路由检查

需要明确：

- 哪些目的地址应该走 `tun`
- 哪个网关或接口负责转发
- 是每台主机加静态路由，还是由中心节点汇聚

若用户说“设备在不同网段，但 app 想像局域网那样直连”，回答里必须出现“目的流量如何进入 `tun`”。

## 4. 防火墙与端口

需要检查：

- underlay 使用的监听端口是否放行
- `tun` 路径中的业务端口是否放行
- SOCKS5 服务监听端口是否允许本地 app 或指定网段访问
- 是否存在本地安全软件或云安全组限制

“隧道建立了但业务不通”时，防火墙与端口是独立排查项。

## 5. MTU 与分片

`tun` 场景经常引入额外封装。
如果出现“能连但吞吐低”“小包正常、大包失败”“TCP 卡顿、UDP 异常”，要把 MTU 作为显式排查项。

## 6. DNS / 服务发现 / 代理接入

即使路由正确，应用也可能因为接入方式错误而表现为“连不上”。
回答时需要区分：

- 连接失败是传输层不通
- 名字解析错误
- 代理没有生效
- 应用依赖广播发现，但 `tun` / 路由方案只提供三层可达

## 7. 推荐输出格式

在方案题里，把“宿主机前置条件”单列成一节：

- 路由
- 转发
- 防火墙
- MTU
- DNS
- 代理入口

这样可以避免把问题都归咎于 `cyfs_gateway.yaml`。
