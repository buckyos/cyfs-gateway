# 场景模板

本文件用于把“业务目标”映射成基于当前实现证据的回答结构。
它不是产品能力清单；凡是超出 [implementation-checked.md](implementation-checked.md) 的部分，都只能写成外部前提、待验证项或保守工程假设。

## 1. 点到点显式地址互通

适用目标：

- 两台设备不在同一网段
- 应用通过固定 IP / 端口或静态域名连接
- 用户接受额外宿主机动作

推荐分析框架：

- 先写已确认的 `tun` 字段事实：`bind`、`mask`、`mtu`、`tcp_timeout`、`udp_timeout`、`hook_point`
- 再写需要补充确认的 underlay、静态路由、防火墙、MTU、是否需要 NAT
- 最后写验证顺序

回答时至少给出：

- 已确认的 `tun` 字段与默认值
- 外部前提
- 应用接入方式
- 验证步骤

常见遗漏：

- 把 `tun.bind` / `mask` 直接写成“已经形成可用私网”
- 没把防火墙、静态路由、MTU 单列为外部前提
- 没说明应用访问的是哪个地址，或是否其实更适合 `socks`

## 2. 多节点地址规划分析

适用目标：

- 多个节点需要显式地址规划
- 用户想讨论中心转发、站点互通或多节点接入

推荐分析框架：

- 先声明：当前 skill 不把 hub-spoke / full-mesh 当成 `cyfs-gateway` 原生能力承诺
- 再分别写 `tun` 基础字段、中心节点职责、边缘节点职责、外部路由与验证步骤
- 若缺少外部前提，就明确写 evidence 不足

回答时至少给出：

- 哪些是 `tun` 基础字段事实
- 哪些是外部组网动作
- 哪些结论仍需实测
- 风险与限制

## 3. 基于 process chain 的数据转发 / 上游转发

适用目标：

- 希望按请求条件、协议探测结果或入口类型，把流量交给某个 server 或某个 upstream
- 目标可能是 HTTP 反向代理、TCP / UDP / RTCP 转发、多上游转发
- 不一定涉及 `tun` 地址规划

推荐分析框架：

- 先判断目标是 server id 还是 upstream URL
- 若目标是 server 对象，用 `call-server`
- 若目标是 URL，用 `forward`
- 若是应用代理接入，切换到 `socks` server 讨论
- 若是 `tun` 地址承载或宿主机路由问题，切换到 `tun + 宿主机路由` 讨论

回答时至少给出：

- 转发入口属于 `call-server`、`forward`、`socks` 还是 `tun`
- 目标对象是什么
- 是否需要多上游与选择算法
- 命中条件写在 stack 还是 server 的 `hook_point`
- 验证顺序

## 4. 应用代理式远端访问

适用目标：

- 只要求某些 app 访问远端服务或远端网络资源
- app 支持 SOCKS5 或系统代理
- 不要求整机进入同一虚拟网段
- 不要求广播发现

推荐分析框架：

- 明确代理入口是本地 app 配代理，还是系统代理转发到 socks 服务
- 明确目标服务范围是固定目标、按域名规则、还是按 PAC / 规则文件选择
- 判断是否需要 `enable_tunnel`
- 判断是否需要 `rule_config`
- 明确这只是应用流量接入，不是整机网络接入

回答时至少给出：

- 哪些 app 可以使用该方案
- 代理目标 `target` 是什么
- 是否需要 `enable_tunnel`
- 规则文件从哪里加载
- 哪些流量不会被这个方案覆盖

## 5. “像同一局域网一样”的 App 互通

先判断 app 的连接方式：

- 若 app 支持手工指定对端 IP / 域名 / 端口：
  先回到 `tun` 的基础字段和外部前置条件
- 若 app 支持 SOCKS5 或可被系统代理接管：
  可按 `socks5` 型代理接入方案设计
- 若 app 依赖广播发现对端：
  先标记为能力边界问题，不能直接承诺

建议输出：

- 该 app 属于“固定地址接入”“代理接入”还是“自动发现接入”
- 如果可固定地址接入，哪些 `tun` 字段已确认、哪些宿主机条件还未确认
- 如果可代理接入，如何配置代理入口与目标规则
- 如果依赖广播发现，需要补什么额外证据或替代方案

## 6. 不同网段设备组成“一个局域网”的推荐口径

推荐口径：

- 默认把它解释为“用户想要显式地址互通，需要先核对 `tun` 的基础字段与外部前提”
- 如果用户说的其实只是“让几个 app 访问远端服务”，切换到 SOCKS5 代理接入
- 如果用户继续强调“像交换机下一样自动发现”，再切换到能力边界说明

推荐回答结构：

1. 目标归类
2. 已确认的 `tun` / `socks` 基础事实
3. 外部前提
4. 验证步骤
5. 风险与限制

## 7. 典型方案骨架

当 evidence 不足以给出精确产品事实时，可以使用下面的保守骨架：

- underlay：
  使用已支持的 `tcp` / `udp` / `tls` / `quic` / `rtcp` 中适合当前环境的 stack
- `tun`：
  只说明 `bind` / `mask` / `mtu` / timeout / `hook_point` 这类已证实字段
- proxy access：
  使用 `socks` server 承载应用代理接入
- policy：
  用 `hook_point` / process chain 控制流量进入目标 server 或上游
- discovery：
  若 app 支持固定地址，则优先使用静态 IP 或 DNS
  若 app 支持代理，则优先使用代理接入
  若 app 依赖广播，则注明当前 skill 无法直接证明可行

## 8. `tun` 问题的固定输出模板

如果用户明确问的是 `tun stack` 怎么用，回答至少按下面顺序展开：

1. 目标归类：
   是字段说明、单机地址承载，还是外部组网讨论
2. 已确认字段：
   `bind`、`mask`、`mtu`、`tcp_timeout`、`udp_timeout`、`hook_point`
3. 已确认默认值：
   `mask = 255.255.255.0`、`mtu = 1500`、`tcp_timeout = 60`、`udp_timeout = 60`
4. 外部前提：
   underlay、静态路由、IP 转发、防火墙、MTU、是否需要 NAT
5. 应用接入：
   应用访问哪个地址、是否应改走代理
6. 验证步骤：
   先验证基础连通，再验证地址与端口，最后验证应用协议
7. 能力边界：
   如果用户期望广播发现、二层互通或原生私网承诺，明确说明当前 skill 不能直接承诺

如果用户要求“给出一台机器上的 `tun` 配置”，答案至少包含：

- `protocol`
- `bind`
- `mask`
- `mtu`
- `tcp_timeout`
- `udp_timeout`
- `hook_point`

额外约束：

- 不允许给出缺少 `hook_point` 的最小示例
- 必须明确 `hook_point` 返回的是 `call-server` 还是 `forward`
- 若当前 evidence 不足以给出完整可运行的 `hook_point` 业务逻辑，必须明确说明还缺少对应的 server 或 upstream 目标
